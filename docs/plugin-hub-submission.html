PB Tracker Sync — Plugin Hub Submission Plan

Concrete steps to get plugin/ ready for RuneLite Plugin Hub peer review.

Submission gate

We do not open the Plugin Hub PR until plugin-side testing passes locally (gradle run, logging in, generating/loading PBs, confirming syncs fire and the panel behaves correctly). Formal backend hosting, database choice, and server-side test coverage are explicitly out of scope for this gate — development targets a local backend (http://localhost:3000) the whole time. Plugin Hub review only looks at the plugin's own code and behavior anyway, not whatever backend a user points it at.

2. Extracting the plugin into its own repo

  1. Create a new public GitHub repo (e.g. pb-tracker-sync-plugin) from the official template: runelite/example-plugin.
  2. Copy everything under plugin/ in this repo into it: build.gradle, settings.gradle, runelite-plugin.properties, LICENSE, README.md, and src/.
  3. Push to main, confirm gradle build passes clean in the new repo (no relative paths back into the old monorepo).
  4. Note the resulting repo's HTTPS clone URL — needed for the manifest in step 3 below.

3. Opening the Plugin Hub PR

  1. Fork runelite/plugin-hub.
  2. Create a branch, add a manifest file under plugins/pb-tracker-sync containing: 
    repository=https://github.com/<you>/pb-tracker-sync-plugin
commit=<full 40-character commit hash>
  3. Push and open a PR against runelite/plugin-hub.
  4. CI runs automatically. If it fails on a build error, fix it in the plugin repo, then update the commit= hash and push again — multiple iterations within the same PR are normal and expected.

4. What gets reviewed (and what doesn't)

Per RuneLite's own Plugin Hub Review docs, the process focuses on exactly two things:

  • Security — no reflection abuse, no native code, dependency scanning on anything beyond the standard RuneLite/OkHttp/Gson stack we already use.
  • Game rule compliance — doesn't reproduce a previously-rejected or rolled-back feature, doesn't violate Jagex's rules.

Explicitly not reviewed: whether the plugin is useful, performant, bug-free, or factually accurate. That's on us to get right independently — it just won't block the PR.

As of April 2026, RuneLite runs an automated review bot capable of auto-approving simple, low-risk updates; anything more involved still goes to a human. New third-party dependencies always require manual vetting and add meaningfully to turnaround — this plugin only uses RuneLite's already-bundled OkHttp + Gson, so that shouldn't apply to us.

5. Timeline expectations

RuneLite doesn't publish a fixed SLA — official guidance is just "be patient." Realistically: anywhere from a few days to a few weeks for a plugin with no novel dependencies and clean CI. Worth a sanity check in the RuneLite Discord for current queue depth before opening the PR.

6. Sync auth (resolved)

/api/sync previously had no authentication — anyone who knew a server's URL could POST a fake PB under any display name. This is now fixed: the plugin generates a random per-install secret on first run (stored via ConfigManager under the pbtracker group, never shown in the UI) and sends it with every sync. The backend binds that secret to an account hash the first time it's seen, and rejects (409) any later sync for the same account hash presenting a different secret — plus a 30-requests/10-minute rate limit per account hash. This isn't full identity proof (nothing is, without Jagex OAuth for third parties) — whoever syncs an account hash first claims it — but it closes the "anyone can spoof anyone" gap. Not a Plugin Hub blocker either way, since review doesn't audit whatever backend a user points the plugin at.
